# 错误处理交互逻辑

## 概述

定义智能体在执行过程中遇到错误时的标准处理流程和交互模式。

## 错误处理原则

1. **快速失败**: 遇到不可恢复的错误立即停止，不尝试猜测修复
2. **清晰反馈**: 错误信息必须包含问题描述、影响范围和修复建议
3. **安全回滚**: 修改失败时自动回滚到修改前的状态
4. **上下文保留**: 错误发生时保留足够的上下文信息用于调试

## 错误场景与处理策略

### 后端错误

| 场景 | 错误类型 | 处理策略 |
|------|----------|----------|
| Ruff 检查失败 | 代码规范 | 自动格式化后重新检查，仍失败则列出问题 |
| 测试失败 | 逻辑错误 | 分析失败原因，提供修复建议 |
| 导入错误 | 依赖问题 | 检查 requirements.txt，提示安装缺失依赖 |
| 数据库错误 | 数据问题 | 检查模型定义，验证 Schema 同步 |
| 类型错误 | 类型注解 | 修正类型注解，确保 Mapped 风格一致 |

### 前端错误

| 场景 | 错误类型 | 处理策略 |
|------|----------|----------|
| ESLint 失败 | 代码规范 | 自动修复可修复项，列出需手动修复项 |
| TypeScript 编译失败 | 类型错误 | 修正类型定义，确保接口一致 |
| 构建失败 | 构建配置 | 检查 Vite 配置和依赖完整性 |
| 测试失败 | 逻辑错误 | 分析断言失败原因，提供修复建议 |

### 集成错误

| 场景 | 错误类型 | 处理策略 |
|------|----------|----------|
| API 调用 401 | 认证失败 | 检查 token 有效性，提示重新登录 |
| API 调用 404 | 路由错误 | 检查 API 路径和后端路由定义 |
| CORS 错误 | 跨域问题 | 检查后端 CORS 配置 |
| 数据不一致 | 前后端接口不匹配 | 对比 api.ts 与 schemas.py |

## 错误恢复流程

```
1. 捕获错误 → 2. 分类分级 → 3. 尝试自动修复 → 4. 验证修复 → 5. 如失败则回滚 → 6. 报告用户
```

### 自动修复范围

以下错误可自动修复：
- Ruff 格式化问题: `ruff format backend/`
- ESLint 可自动修复项: `npm run lint -- --fix`
- Import 排序问题: `ruff check --fix backend/`

以下错误需手动修复：
- 逻辑错误
- 类型不匹配
- 缺失的业务逻辑
- 架构设计问题
