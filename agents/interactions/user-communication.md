# 用户沟通交互逻辑

## 概述

定义智能体与用户之间的标准沟通模式，确保信息传递的清晰性和效率。

## 沟通原则

1. **主动确认**: 在执行不可逆操作前确认用户意图
2. **进度透明**: 实时报告任务执行进度
3. **结果导向**: 优先展示最终结果，技术细节按需提供
4. **语言一致**: 使用中文沟通，代码注释遵循项目规范

## 沟通场景

### 场景一：需求澄清

当用户需求不明确时，主动提问：

**触发条件**:
- 需求描述模糊（如"优化一下"）
- 缺少关键参数（如未指定组件类型）
- 存在多种实现方案

**沟通模板**:
```
我理解您想要 [复述需求]。为了确保实现符合预期，需要确认以下几点：
1. [具体问题1]
2. [具体问题2]

建议方案：
- 方案A: [描述] - 优点 / 缺点
- 方案B: [描述] - 优点 / 缺点
```

### 场景二：执行确认

在执行不可逆操作前确认：

**触发条件**:
- 删除操作（文件、数据库记录）
- 数据库结构变更
- 生产环境部署
- 大范围代码重构

**沟通模板**:
```
即将执行 [操作描述]，此操作 [影响说明]。

变更范围：
- [文件1]: [变更内容]
- [文件2]: [变更内容]

确认执行？(y/n)
```

### 场景三：进度报告

长时间任务执行中的进度更新：

**触发条件**:
- 任务包含 3 个以上步骤
- 单步执行时间较长

**沟通模板**:
```
[任务名称] 执行中...

✅ 步骤1: [描述] - 完成
🔄 步骤2: [描述] - 进行中
⏳ 步骤3: [描述] - 待执行
```

### 场景四：结果汇报

任务完成后的结果汇报：

**沟通模板**:
```
[任务名称] 已完成 ✅

变更摘要：
- 新增文件: [文件列表]
- 修改文件: [文件列表]
- 删除文件: [文件列表]

验证结果：
- Lint: ✅ 通过
- 测试: ✅ 通过
- 构建: ✅ 通过

后续建议：
- [建议1]
- [建议2]
```

### 场景五：问题上报

遇到无法自动解决的问题时：

**沟通模板**:
```
执行 [任务名称] 时遇到问题 ⚠️

问题描述：
[具体描述]

已尝试的解决方案：
1. [方案1] - 结果
2. [方案2] - 结果

需要您的决定：
- [选项A]: [描述]
- [选项B]: [描述]
```

## 反馈收集

在关键交付后收集用户反馈：

```
以上实现是否符合您的预期？如有调整需求，请告知：
1. 功能是否符合需求？
2. 代码风格是否一致？
3. 是否需要补充测试？
```
